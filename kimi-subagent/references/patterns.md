# 委派模式：并发、拆分、何时委派

## 何时该委派给 kimi（vs 宿主自己做）

**适合委派**：
- 任务**自包含**、边界清晰，能用一段 prompt 完整描述。
- 需要**隔离上下文**：探索大仓库、跑长任务，不想污染宿主主线对话。
- 可**并行**的独立子任务（多个模块各查各的）。
- 想用 **Kimi 模型特定能力**（如其 Agent/工具链）来完成某段工作。

**不适合委派**：
- 任务高度依赖宿主当前对话上下文、且难以完整转述。
- 一两步就能做完、委派开销（进程启动 + 上下文转述）不划算。
- 需要与用户频繁交互确认的任务（headless 无人交互）。

## 并发委派

把彼此**无依赖**的子任务拆开，并行启动多个 `kimi -p`，全部完成后宿主汇总。

```bash
# 三个独立子任务并行；各自落盘，wait 后汇总
kimi -p "$T1" --output-format stream-json --auto -w /repo 1>r1.jsonl 2>/dev/null &
kimi -p "$T2" --output-format stream-json --auto -w /repo 1>r2.jsonl 2>/dev/null &
kimi -p "$T3" --output-format stream-json --auto -w /repo 1>r3.jsonl 2>/dev/null &
wait
# 读取 r1/r2/r3 末行结论，宿主合并
```

要点：
- **控制并发度**：按机器资源限制同时运行的实例数（如一批 3–5 个），避免过载。
- **隔离写入**：并行实例若都写文件，给不同 `-w` 工作目录或不同输出路径，避免互相覆盖。
- **失败重试**：对退出码非 0 的子任务单独重跑；重试前检查 prompt 是否上下文不全。

## 任务拆分原则

- 一个子任务 = 一段能独立验收的工作，**显式声明期望产出**（文件清单 / 一段 JSON / 结论）。
- 拆分粒度匹配可并行性：能并行就拆细，有依赖就串成多轮（见下）。
- 跨子任务的共享信息（约定、命名、接口）**在每个 prompt 里各自重述**——子 agent 之间不共享上下文。

## 多轮 / 流水线委派

有依赖的步骤用同一子会话续接，让后续轮次保留前文：

```bash
# 轮 1：探索并产出方案（拿到 session id 见 headless-output.md）
kimi -p "在 /repo 探索鉴权模块，输出重构方案" --output-format stream-json --auto -w /repo
# 轮 2：基于上一轮结论实施
kimi -S <id> -p "按上一轮方案实施重构，最后输出改动文件清单 JSON"
```

或宿主充当编排器：轮 1 的结论由宿主提取后，**写进**轮 2 新实例的 prompt（无需共享会话）。

## 汇总与校验

- 宿主收集各子 agent 的结论后**自行校验**（读改动文件、跑测试），不要盲信子 agent 自述「已完成」。
- 对关键改动，可再派一个 kimi 以**只读**取向复核（prompt 明确「只读分析、不改文件」）。

## 与内置子 agent 的关系

Kimi Code CLI **内部**已有内置子 agent（`coder` / `explore` / `plan`），由 kimi 主 agent 自动派发、上下文隔离。本 Skill 讲的是**宿主从外部**把整个 `kimi` 进程当子 agent 调用——两层不冲突：你 `kimi -p` 派出去的那个 kimi，内部仍可能再自行派发它的内置子 agent。
